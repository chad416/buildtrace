## Supported locales

BuildTrace Beta supports these locales from day one:

- en - English
- cs - Czech
- sk - Slovak
- pl - Polish
- de - German
- fr - French
- es - Spanish

## Locale routing

The web app uses locale-based routes:

- /en
- /cs
- /sk
- /pl
- /de
- /fr
- /es

The root route redirects to `/en`.

Unsupported locale routes are validated and should return `notFound()` according to the current route rules.

## Translation rule

No user-facing UI text should be hardcoded in product screens.

User-facing text should use translation keys.

This includes:

- navigation
- buttons
- form labels
- validation messages
- empty states
- loading states
- error states
- status labels
- document category labels
- QR portal labels
- emails
- PDF export labels
- language switcher labels
- app shell labels
- landing page content
- footer links and labels
- page-shell titles
- page-shell descriptions
- page-shell empty-state content

## Language switcher behavior

The current language switcher:

- renders one link per supported locale
- uses labels from translation message files
- uses the shared `locales` constant from `@buildtrace/i18n`
- marks the current locale with `aria-current="page"`
- uses locale routes such as `/en`, `/cs`, `/de`, and `/es`
- does not hardcode visible language labels inside the component
- is placed in the translated app shell header

Manual runtime verification passed:

- localized page loaded
- header rendered
- footer rendered
- language switcher was visible in the header
- switching from Spanish to English worked
- supported locale routes loaded without crash

## I18n boundaries

BuildTrace UI is multilingual.

Uploaded documents are not auto-translated.

Generated PDF summaries may later use UI translation keys, but source document contents remain in their original uploaded language.

QR portal language detection is not implemented yet.

Customer preferred language handling is not implemented yet.

No language-specific access-control rules are implemented yet.
