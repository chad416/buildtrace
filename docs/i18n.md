## Locale routing

The web app uses locale-based routes:

- /en
- /cs
- /sk
- /pl
- /de
- /fr
- /es

The root route redirects to `/en`.

Unsupported locale routes are validated and should return `notFound()` according to the current route rules.

## Translation rule

No user-facing UI text should be hardcoded in product screens.

User-facing text should use translation keys.

This includes:

- navigation
- buttons
- form labels
- validation messages
- empty states
- loading states
- error states
- status labels
- document category labels
- QR portal labels
- emails
- PDF export labels
- language switcher labels
- app shell labels
- landing page content
- footer links and labels
- page-shell titles
- page-shell descriptions
- page-shell empty-state content
- dashboard placeholder card labels
- login placeholder copy
- settings placeholder section labels

## Language switcher behavior

The current language switcher:

- renders one link per supported locale
- uses labels from translation message files
- uses the shared `locales` constant from `@buildtrace/i18n`
- marks the current locale with `aria-current="page"`
- uses locale routes such as `/en`, `/cs`, `/de`, and `/es`
- does not hardcode visible language labels inside the component
- is placed in the translated app shell header

Manual runtime verification passed:

- supported locale routes loaded without crash

## Styling and CSS pipeline note

Phase 1 uses Tailwind utility classes for the industrial UI shell.

A plain-HTML browser rendering issue was found during manual browser testing. The UI had Tailwind classes in JSX, but the Tailwind/PostCSS CSS pipeline was not fully wired.

Root cause:

- the web UI used Tailwind utility classes
- the app needed a global stylesheet imported by the localized layout
- Tailwind/PostCSS dependencies and configuration were missing or incomplete
- browser output initially appeared unstyled/plain HTML

Fix completed in the Phase 1 industrial polish chunk:

- added `apps/web/src/app/globals.css`
- added `apps/web/postcss.config.mjs`
- imported `globals.css` from the localized layout
- added required Tailwind/PostCSS development dependencies for the web app
- updated `pnpm-lock.yaml`

The global CSS uses:

```css
@import 'tailwindcss';
```

Manual verification confirmed:

- Next dev served a real CSS asset under `/_next/static/chunks/`
- the CSS asset was non-empty
- Tailwind classes such as `bg-neutral-950` and `text-stone-50` were present
- browser pages rendered as styled shell pages, not plain HTML

## I18n boundaries

BuildTrace UI is multilingual.

Uploaded documents are not auto-translated.

Generated PDF summaries may later use UI translation keys, but source document contents remain in their original uploaded language.

QR portal language detection is not implemented yet.

Customer preferred language handling is not implemented yet.

No language-specific access-control rules are implemented yet.

No authentication, database, storage, QR portal, ticket backend, CRUD, or real dashboard data has been added as part of Phase 1 shell polish.
